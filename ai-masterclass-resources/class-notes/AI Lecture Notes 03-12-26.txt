Why not one agent?

"You could build a single agent that captures frames, describes them, embeds them, scores the results, and adjusts its approach. But remember what we said about monolithic applications — they don't scale. A single agent with too many jobs becomes a jack of all trades." ³

     ↓

     ↓

     ↓

Section 3: Skill 2 — Orchestration (0:35-0:55)

How do agents coordinate? Message bus, not method calls.
0:35-0:38 — Pipeline vs. Conversation

"Before we look at the code, there's a design decision you need to make with any multi-agent system: are your agents a pipeline or a conversation?"

Agent Pipeline:

    Each agent does one step, passes the output forward to the next
    Like a factory assembly line: capture → embed → score → refine
    Agents don't know who's upstream or downstream — they just read from one stream and write to another
    Best for: problems with clear stages and predictable throughput

Agent Conversation:

    Agents talk directly to each other by name — "@embedder, can you try a different approach on this frame?"
    Like a group chat: any agent can message any other agent
    A central router (or gateway) delivers messages to the right recipient
    Best for: open-ended collaboration, negotiation, ambiguous tasks where agents need back-and-forth

"swarm-v2 is a pipeline. The work has a clear sequence: capture a frame, propose how to embed it, test that approach, score it, refine. Each step feeds the next. Nobody needs to go back and ask clarifying questions. That's what makes a pipeline the right fit."

Speaking note: "The pipeline vs. conversation distinction is the first question to ask when designing a multi-agent system: does my problem have a clear sequence of steps, or do agents need to negotiate and go back and forth? If it's sequential, use a pipeline. If it's collaborative, use a conversation. Most real systems end up being a pipeline with a conversation layer on top for exception handling — but start with the simpler pattern."
0:38-0:42 — The Message Bus Pattern

"So we're building a pipeline. The agents communicate through Redis Streams — a built-in feature of Redis that acts as a message bus."

Why a message bus instead of direct function calls?

    Agents can run in different processes, different machines, different languages
    If one agent is slow or crashes, the others keep working
    You can scale by adding more consumers to a stream (Redis distributes the work automatically)
    The message history is persistent — you can replay it, audit it, debug it

The three streams:
Stream 	Producer 	Consumer 	Message
screen:frames 	FrameCaptureAgent 	EmbedderAgent 	Batch of screen frames (up to 15 unique per 30s window)
screen:approaches 	EmbedderAgent 	ApproachTesterAgent 	Proposed embedding strategy
screen:results 	ApproachTesterAgent 	RefinerAgent + Monitor 	Score + impact_items + top_positive/top_negative

"Plus agent:health — every agent publishes a heartbeat every 20 seconds so the monitor knows who's alive."

Speaking note: "Consumer groups are Redis's way of splitting work across multiple agents. XREADGROUP means if you run three embedder instances, Redis automatically gives each one different messages — no two agents process the same frame. ⁸ Think of it like a restaurant ticket wheel: the kitchen puts tickets up, and whichever cook grabs it first handles that order."
0:42-0:48 — The Message Schema

"Every message on the bus follows the same shape:"

{
  "id":        "msg_a3f2c1",
  "from":      "embedder",
  "to":        "approach-tester",
  "type":      "approach_proposal",
  "payload":   { ... },
  "context":   { "agent": "embedder", "score": 0.68, "top_positive": "...", "top_negative": "..." },
  "timestamp": "2026-03-12T18:30:00Z"
}

"Notice the context field. Every message carries the sender's current status — its current score, top positive observation, and top negative. So when the approach-tester receives a proposal, it doesn't just see the proposal. It sees the embedder's performance evidence. That's how agents share awareness without a central coordinator."

The Agent Help Loop:

1. OBSERVE    — Batch 30s of screen activity (up to 15 unique frames)
2. PROPOSE    — What approach might improve the embedding?
3. SCORE      — Two-pass evaluation:
                 Pass 1: Itemize observations (+/- 1, 2, 3, or 5 impact)
                 Pass 2: net_impact / √(items) → normalized score
4. REINFORCE  — Feed winners back. Discard losers. Tune capture params.

"This loop runs continuously. Batch → Propose → Score → Reinforce. Every cycle, the system has more evidence about what works and what doesn't — and the refiner can tune not just the embedding approach, but the capture parameters too."

Section 5: Skill 4 — Evaluation (1:15-1:30)

Honest scoring is the whole game.
1:15-1:20 — Why Eval Is the Hardest Part

"If monitoring is the eyes, evaluation is the brain. The quality of your scoring determines whether the swarm improves or just spins."

"Here's the trap: if you let an LLM score its own work with a single 'rate this 1-10' prompt, you'll get inflated, inconsistent scores every time. The LLM wants to be helpful. It'll say 7/10 when the real answer is 4." ¹⁰

Callback to Week 5: "Remember what we talked about with the vibeathon judging pipeline? Don't ask for one big holistic score. Break it into smaller, bounded decisions the LLM can make accurately, then aggregate mathematically. Same principle here."
1:20-1:25 — Three Eval Patterns

Open skill-4-eval/eval.py and walk through:

Pattern 1: Two-Pass Rubric Eval
Pass 1 (LLM): Itemize observations across four dimensions, each with an impact score (+/- 1, 2, 3, or 5):

    intent_capture — does it capture what the person is trying to accomplish?
    specificity — specific enough to distinguish similar-looking frames?
    cognitive_state — does it infer the user's mental state or work stage?
    noise_resistance — avoids irrelevant visual details?

Pass 2 (math): net_impact / √(total_items) * 8.0, centered on 50, scaled to 0-1. The √ means quality beats quantity — five items averaging +3 beats twenty items averaging +1.

"The rubric is the spec. Same as TDD — you define what 'good' looks like before you evaluate. But the key insight is separating the LLM's job (itemize what you see) from the math (derive a score). The LLM is bad at assigning a single 0-1 number. It's great at listing specific observations." ⁶

Pattern 2: Comparative Eval
Given two approaches to the same frame, which is better and why?

"This is powerful because relative judgments are easier for LLMs than absolute scores. 'Is A better than B?' is a simpler question than 'How good is A on a scale of 0-1?' Use this when you need to choose between competing approaches."

Pattern 3: Regression Check
Did the swarm get worse? Compare average scores from the last N rounds to the previous N rounds.

"This catches the case where a refiner proposes a new approach that sounds smart but actually degrades quality. Without regression checks, you can lose ground without knowing it."
1:25-1:30 — The Scoring Schema

"Every eval in this system produces the same two-pass schema:"

# Pass 1 — LLM itemizes observations
"items": [
    {"dimension": "intent_capture",  "observation": "Captured the pivot table workflow", "impact": +3},
    {"dimension": "noise_resistance", "observation": "Described window chrome instead of task", "impact": -2},
    {"dimension": "specificity",      "observation": "Distinguished this Excel session from yesterday's", "impact": +2},
]

# Pass 2 — deterministic math (no LLM)
"derived": {
    "overall":            0.607,   # net_impact / √(items) → normalized 0-1
    "net_impact":         3,
    "total_items":        5,
    "normalized_impact":  1.342,
    "raw_score":          60.7
}

"Pass 1 is the LLM's job: list what you see, score each observation's impact. Pass 2 is deterministic math — derive_score() is a pure function, no LLM involved. The agent sees the top positive and top negative observations injected into its system prompt. It literally reads 'Best: Captured the pivot table workflow | Worst: Described window chrome instead of task' and adjusts."

"This is reinforced learning without retraining. No fine-tuning, no gradient descent, no GPU cluster. Just itemized evidence fed into the next prompt." ¹ ⁶

Speaking note: "Emphasize: the impact items are doing the job that a loss function does in traditional ML. They're the signal that tells the system which direction to adjust. But it's natural language observations with explicit magnitude, not opaque gradients. And the math is trivially auditable — you can look at the items and verify the score makes sense. That's what makes this accessible to developers who don't have ML backgrounds."
Section 6: Skill 5 — Refining (1:30-1:45)

Closing the loop: how the swarm gets smarter.
1:30-1:37 — The Refiner Agent

Open skill-5-refining/refiner.py:

"The refiner is the agent that makes the whole system a learning system instead of just a processing pipeline. Without it, you have agents that capture, embed, and score — but never improve. The refiner closes the loop."

How it works:

    Reads score history from screen:results (every N scores, not every frame — slower cadence)
    Groups impact items by dimension, sorts by magnitude — high-impact observations (+3/+5 and -3/-5) surface first
    Proposes a refined approach or reinforces the current one
    Can also tune capture parameters — batch size, snap frequency, dedup threshold
    Publishes the new approach back to a Redis key that embedder agents watch

"The refiner plays the role of the human editing program.md — but it does it automatically, based on scored evidence. This is the Karpathy pattern: ¹ you're not retraining the model. You're updating the context that shapes its behavior." ¹¹

Walk through the code:

# The refiner reads impact items across multiple scores:
"Looking at your last 5 scored results:
 - Top positives (recurring):
   +3 intent_capture: 'Captured the pivot table workflow'
   +3 cognitive_state: 'Inferred user was comparing data sources'
 - Top negatives (recurring):
   -2 noise_resistance: 'Described window chrome instead of task'
   -3 specificity: 'Description could match any spreadsheet session'

 Proposed refinement:
 ALWAYS include: (1) what tool, (2) what task, (3) what cognitive state,
 (4) what work stage, (5) what intent. Drop: visual descriptions of UI layout.
 Also: increase batch window to 45s for spreadsheet tasks — longer context helps."

"See how the refiner synthesizes across multiple rounds? It's not just reading one score — it's finding patterns in the impact items. 'Described window chrome' keeps appearing as a -2? The refinement will push toward task-level descriptions. And notice it can also tune the capture parameters — if longer batch windows produce better scores for certain activities, the refiner adapts how the system observes, not just what it describes."

Section 7: Connecting It All — The AI-Native Checklist (1:45-1:55)

How to know when something is truly AI-native
1:45-1:50 — Seven Properties

"We've spent an hour inside a real AI-native system. Now let's step back and name what makes it AI-native — not AI-enhanced, not AI-cosmetic." ¹²

Walk through each property, pointing back to swarm-v2 as the example:

    Natural language is a first-class interface ¹ — Every agent communicates through natural language prompts and responses. The message bus carries descriptions, approaches, and scores — all in English. No rigid API schemas for the reasoning layer.

    Behavior is probabilistic, not deterministic ¹³ — Run the swarm twice, you'll get different scores and different approaches. That's not a bug. The guardrails (the scoring rubric, the product spec, the 12-factor principles) keep it bounded. The variability is where the learning happens.

    Agents replace workflows ⁴ — The embedder doesn't follow a decision tree. It reasons about each frame, consults its memory, and proposes a novel approach. The approach-tester doesn't check boxes — it evaluates quality with judgment.

    Context is the new code ⁷ — The system prompt evolves based on score history. Nobody changed the code between round 1 and round 3. The context changed. That's what changed the output quality.

    Multi-model orchestration is built in ¹⁴ — Ollama does the reasoning; Hindsight handles embedding and similarity search. Three different models, each doing what it's best at.

    The autonomy slider exists — Run in demo mode (synthetic frames) vs. live mode (real screen capture). Run one agent manually vs. the full swarm autonomously. Dial the refiner cadence up or down. The system is designed for variable autonomy.

    It learns and adapts ¹ ⁶ — Scores go from 0.45 to 0.82 across three rounds. The refiner synthesizes patterns and updates the approach. No retraining. No fine-tuning. Just scored feedback fed back into context.

Instructor Notes
Pacing Guidelines

    If running behind: Shorten Section 4 (monitoring) to 5 min — just show the dashboard, skip API endpoints. Shorten Section 7 (AI-Native Checklist) to 5 min — the students already lived through the seven properties, don't need to lecture them again.
    If running ahead: Let the swarm run longer during demos. Deeper discussion at the end. Let students ask questions about the code during each section instead of holding for the end.
    The break should land around the 55-minute mark. If you're early, extend the orchestration demo. If you're late, shorten the break to 5 minutes.

Demo Risk Mitigation

    Redis/Docker fails: Have a backup plan — python agent.py (Skill 1) doesn't need Redis. You can teach agent definition, the 12-factor principles, and the learning loop with just Skill 1. The orchestration and monitoring become slides-only.
    API key issues: Test before class. Have a second API key ready. Pre-run the demo and save terminal output as screenshots in case of live failure.
    Slow inference: Demo mode uses synthetic frames (no real screen capture). If inference is slow, reduce the swarm to 2 agents (capture + embedder) and hand-score the results to show the pattern.

Common Questions to Anticipate

    "Why Redis Streams?" — It's built into Redis, so there's nothing extra to install — one Docker command and you have a message bus. For a single-user system like this, it's more than enough. If you outgrow it, the pattern (publish to a stream, consume from a stream) transfers to any message bus technology.
    "Why not use a framework like the Claude Agent SDK for this?" ¹⁸ — You absolutely could. swarm-v2 uses raw HTTP calls to Ollama to keep the agent logic transparent. Frameworks add convenience (tool loops, context management) but hide some of what's happening. For teaching, raw is better. For production, pick the framework that fits your stack.
    "How is this different from fine-tuning?" — Fine-tuning changes the model's weights permanently. This changes the model's context per-call. ¹ Fine-tuning requires training data and compute. This requires scored feedback and good prompts. Both make the system better. This is faster to iterate and cheaper to run.
    "How would you deploy this in production?" — Each agent becomes a container/process. Redis Streams handles the messaging. ⁸ Add auth, rate limiting, error handling, dead letter queues. The architecture is already production-shaped — it just needs hardening.
