SECTION 2: The Greenfield Mindset (0:12–0:27)

Spec-first vs. prompt-first, and the restart instinct
0:12–0:20 — Two Modes, One Spectrum

The core tension on greenfield projects:

    Some devs over-spec and waste the speed advantage AI gives them. Others under-spec and end up with a mess that needs to be thrown away after two weeks.

Both failure modes exist. The skill is choosing the right mode for the moment.

Mode 1: Spec-First (Deliberate)

Before any code is written, you define exactly what you're building:

    Product vision → data architecture → design system → section-by-section requirements
    Tools for this: Design-OS (Brian Casel / Builder Methods), superpowers:brainstorm (Claude Code plugin)
    This is the "human-driven" part of human-driven AI-powered development — you're steering, AI is executing

When spec-first wins:

    You know the domain well (you've built something like this before)
    You're working with other developers (shared spec = shared understanding)
    The project has regulatory or compliance requirements
    You need a specific UX — the product has opinions

Mode 2: Prompt-First (Exploratory)

Start generating and learn from what you see:

    Write a rough prompt describing the idea → let AI build → evaluate the output → decide what you actually want
    This changes how you THINK about the implementation in ways that require starting over with better context
    3–4–5 complete restarts in a fraction of the time it would take to build one careful version

When prompt-first wins:

    You're not sure what the product should be yet
    You're building alone or with a very small team
    Speed of learning > quality of output
    It's a landing page, prototype, or internal tool

Meta prompting — a prompt-first power move:

One underused technique in prompt-first mode: get the AI to write the prompt for you. Describe your rough idea in plain language, then ask Claude: "Write me a detailed prompt I can use to build this." The AI knows what context it needs and will ask for or assume things you wouldn't think to specify. The resulting prompt is often 10x more effective than what you'd write yourself.

This compounds fast: meta prompt → first build → evaluate output → refine the meta prompt → better build.

The key insight: These aren't opposing philosophies — they're different tools. Most good greenfield projects use BOTH: prompt-first to discover what you're building, spec-first to build it properly once you know.
0:20–0:27 — The Restart Instinct

Name it explicitly — because it feels wrong:

Every experienced developer has a deeply ingrained instinct: if something isn't working, debug it. Find the root cause. Fix it. Don't throw away work.

In AI-assisted development, this instinct is often wrong.

    "Sometimes the faster path is not 'debug the AI's bad output' but 'regenerate everything with better context.'"

Why this is true on greenfield:

    The longer a bad AI implementation runs, the more debt accumulates
    AI doesn't have emotional attachment to its output — it regenerates instantly
    A regenerated version starts clean; a debugged version carries the scar tissue of the previous wrong direction

How to recognize the restart moment:

    You've spent >10 minutes trying to fix something the AI got fundamentally wrong
    The AI's approach doesn't match your mental model of the architecture
    You're in a "duct tape on duct tape" spiral — each fix requires another fix
    The code is "working" but you don't understand why

The practical move: Hit Escape. Open a new context window. Paste your notes on what went wrong. Say: "Here's what I tried and why it didn't work. Let's approach this differently." Regenerate.

Git worktrees — the restart instinct at the filesystem level:

Claude Code's --worktree flag (and vanilla git worktree) lets you run multiple implementations simultaneously in separate working directories — same repo, different branches, each with their own file state. No stashing, no branch-switching, no messing up a working experiment to try something else.

Use case: You've got a prompt-first build going, but you're not sure the architecture is right. Spin up a second worktree, try a completely different approach, compare. Keep the best one. Neither approach touches the other.

# Claude Code
claude --worktree   # opens in a new isolated worktree

# Or vanilla git
git worktree add ../my-project-alt experiment-branch

This connects to Week 2's parallel context windows: Restart doesn't mean restart your session or your project — it means open a clean context window (or a clean worktree) with better information and try again.

0:32–0:40 — Design for Products: The Spec-Driven Approach

The Design-OS framework (Brian Casel / Builder Methods):

For anything beyond a landing page, the missing step is designing BEFORE you build. Design-OS is exactly that — a free, open-source, MIT-licensed workflow that takes you from product idea to a complete handoff package through four phases:

    Product Planning — vision, roadmap, data architecture (/product-vision, /data-architecture)
    Design System — colors, typography, application shell (/design-tokens, /design-shell)
    Section Design — requirements, sample data, UI for each feature (/shape-section)
    Export — complete handoff package ready for AI implementation

Each command initiates a CONVERSATION — not a one-shot generation. You answer questions, validate decisions, approve the design at each step. The output is a design document that AI agents reference throughout the entire project.

Why this matters:

AI agents struggle when you ask them to simultaneously determine WHAT to build AND build it. Design-OS separates those concerns. You determine what to build (with AI's help, through structured conversation). Then AI builds it with precision.

This is spec-driven development applied to design decisions.

superpowers:brainstorm (Claude Code plugin) operates at the feature level:

    Instead of asking Claude to build a feature, :brainstorm first asks clarifying questions to understand what you actually need, explores design alternatives, and presents the emerging approach in sections for your approval — before saving a complete design spec.

Install: /plugin install superpowers in Claude Code. Then use /superpowers:brainstorm [rough idea] before any feature implementation.

Together: Design-OS for product-level design → superpowers:brainstorm for feature-level clarification → AI implementation.
0:40–0:47 — Component Libraries as AI Foundation

The fundamental shift: code you own vs. packages you import

Traditional libraries: npm install @mui/material → black box → AI can't see or modify internals.

shadcn/ui: npx shadcn-ui@latest add button → source code copied into your project → AI can read, understand, and modify it directly.

This is why shadcn/ui is the right foundation for AI-assisted greenfield development. When code lives in your project, AI has perfect context — it sees how components are used elsewhere, understands your patterns, and generates new components that fit.

The critical first step: customize before you build

Default shadcn ships with neutral grays and Vercel-inspired spacing. Don't ship a product with default shadcn — it screams "I used a template."

Before writing a single component:

    Run npx shadcn-ui@latest init (configures components.json)
    Define your design tokens in globals.css — colors, typography, spacing, border radius — as CSS custom properties under :root and .dark
    Set cssVariables: true in components.json (consistency; pick one approach and stick with it)
    Install only your base 3–4 components first (Button, Card, Input, Dialog) and verify theming works end-to-end
    Commit this as your design system baseline BEFORE generating components with v0 or AI

Use tweakcn.com for an interactive CSS variable editor, or copy from ui.shadcn.com/themes.

Why this matters for AI tools: When globals.css and components.json are clean and opinionated, Claude and Cursor understand your design system immediately. They'll suggest consistent variants and won't hardcode color values.

The greenfield design toolkit — in order of setup:

    Customize tokens first → set up globals.css design tokens before building anything (tweakcn.com, ui.shadcn.com/themes)
    Tailwind users: npx @truefrontier/tailwind-for-agents → configures Tailwind so AI agents read and write it reliably (Kevin's tool)
    Custom shadcn → follow the 5-step baseline above before generating anything
    UI generation options (pick what fits):
        superdesign.dev — open-source VS Code/Cursor extension; generates 10+ parallel UI variants from a prompt, no API key required
        pencil.dev — design canvas inside VS Code; sketch → React/HTML/CSS, Figma import, Git sync; free in early access
        v0 by Vercel → natural language to shadcn/ui component scaffolding
        Figma MCP — official Figma MCP server; AI agents read component trees, layout constraints, and design tokens directly from Figma files (vs. guessing from screenshots)
    shadcn/ui MCP server → gives Claude accurate component knowledge (prevents hallucinating props that don't exist)

Nick Saraev's GEMINI.md prompt (see the Antigravity demo in Section 3) is the current ceiling of one-shot landing page generation. For anything beyond a landing page — state management, data architecture, complex UX — use the toolkit above.

Key setup move: Add the shadcn/ui MCP server to Cursor or Claude Code on every new project. It gives AI real-time access to the component registry — accurate props, proper usage patterns, correct installation commands.

SECTION 4: Testing Phase — Avoiding the Circular Validation Trap (0:57–1:12)

The problem with AI testing AI's own code, and how TDD fixes it
0:57–1:04 — The Circular Validation Problem

The most important testing concept for AI-assisted development:

    When AI writes code AND the tests for that code, the tests only verify that the code does what the code says — not whether it meets actual requirements.

Concrete example:

    An AI builds a tax calculator assuming a 7% US rate. It also generates tests that only verify the 7% calculation. Tests pass perfectly. The app breaks immediately for Canadian users (5% GST) and for any state with additional taxes. The AI never questioned its own assumption, and the tests validated the assumption instead of the requirement.

This is circular validation — a closed feedback loop where the same mind that made the mistake also verifies it.

It's not a bug in the AI. It's a fundamental property of any system that evaluates its own work.

The fix: Spec-first, then verify with a fresh session

The solution isn't to have humans write the tests manually (that's slow and doesn't scale). It's to break the circular loop at the spec level.

The workflow:

    AI writes a spec document first — before any code. A detailed requirements doc covering happy paths, edge cases, business rules, error conditions. Not a test file — a human-readable spec.
    Human reviews the spec line by line — this is the critical step. You're not reviewing code; you're reviewing whether AI's assumptions about what you want are correct. This is fast and catches misaligned requirements before a single line of code is written.
    AI writes code AND tests against the spec — now both are anchored to the agreed-upon spec, not to each other in a closed loop.
    Run a separate post-implementation verification session — open a clean context, give Claude the spec and the running app, and ask it to verify the implementation using Chrome DevTools or Playwright. Fresh context = no anchoring to the original implementation assumptions.

    A fresh session sees the app the way a user does — without the implementation context that might blind the original agent to its own mistakes.

This is more effective than TDD in practice because it scales: the spec review is fast, the separate verification session is independent, and you're not responsible for writing every test by hand.

What AI handles well vs. what humans must own:
AI handles well 	Human must own
Generating spec documents 	Validating spec assumptions
Writing tests from a spec 	Reviewing spec line by line
E2E flows from user stories 	Business rule edge cases
Post-implementation verification 	Security and threat-model scenarios

SECTION 5: Deployment Phase — Right-Sizing from Day 1 (1:12–1:27)

A decision framework for where and how to deploy, and what you must set up before you launch
1:12–1:20 — The 6-Question Decision Framework

The most common greenfield deployment mistake: Picking infrastructure for the company you want to be, not the company you are.

Kubernetes for an MVP is over-engineering. No logging in production is under-engineering.

Work through these six questions before picking a platform:

    What's the timeline to first paying customer?

        <3 months → PaaS (Vercel, Railway, Render)

        3–6 months → start on PaaS, plan for graduation

            6 months → can invest in infrastructure upfront

    What's the compute profile?
        Stateless APIs / functions → serverless
        Long-running processes → containers
        Complex microservices → Kubernetes only if team has real expertise

    How global is the user base?

        <5% outside primary region → single-region PaaS is fine

            30% outside primary region → plan for edge or multi-region

    What's the data access pattern?
        Mostly reads, small data → SQLite (Turso) or PostgreSQL (Neon)
        Mixed reads/writes → PostgreSQL (Neon, RDS, PlanetScale)

    How much DevOps expertise does the team have?
        <1 year → PaaS (Railway, Render)
        1–3 years → containers (ECS, Cloud Run)
        5+ years → Kubernetes only if complexity is worth it

    What's the biggest risk you're trying to avoid?
        Vendor lock-in → containers + IaC + portable databases
        Surprise costs → transparent usage-based pricing
        Operational burden → managed services

The platform landscape in one paragraph:

For most greenfield MVPs in 2026, Railway is the standout recommendation — full-stack (database + backend + frontend in one platform), transparent usage-based pricing, no pricing cliffs. Vercel is excellent for Next.js but has a painful cliff: Free ($0) → Pro ($20/mo) → Enterprise ($20K+/year), with multiple meters compounding costs. One moderately-trafficked app can hit $500–800/month unexpectedly. Render and Fly.io are solid alternatives. Cloud Run is the right move when you need container-level control without Kubernetes complexity. Kubernetes almost never makes sense for a greenfield project unless you have a dedicated platform team.

For databases: Supabase if you want all-in-one (auth + database + storage). Neon for Next.js projects with CI/CD iteration (its branching feature gives you instant database copies per branch). Turso for read-heavy content apps that need global low latency.
1:20–1:27 — Five Things Experienced Devs Must Set Up Before Launch

These are not optional retrofits. Each one is a time bomb if you skip it.

1. Secrets management — not .env files

Plain-text environment variables are not safe. In 2025, GitGuardian tracked a single attack campaign that leaked 3,325 secrets from PyPI, npm, and DockerHub.

Day 1 setup: Doppler for local development (manages .env for your team without committing it), AWS Secrets Manager or HashiCorp Vault for production. Automated secrets scanning in CI with TruffleHog or GitGuardian.

2. Observability via OpenTelemetry

Datadog is powerful and expensive ($600–3,000/month). And it creates vendor lock-in through its proprietary SDK.

Use OpenTelemetry from day 1 — the vendor-neutral standard for instrumentation. Once your code is instrumented with OpenTelemetry, you can send data to any backend (Grafana Cloud, Datadog, Jaeger, etc.) without re-instrumenting. This is your escape hatch.

Minimum stack: Prometheus (metrics) + Grafana (dashboards) + structured JSON logging. Cost: ~$0 self-hosted, $50–100/month managed.

3. CI/CD from commit one

GitHub Actions or GitLab CI. This shouldn't be day 3 — it should be your second commit, right after the "hello world" commit. AI can scaffold the YAML for you in 2 minutes.

4. Database backup strategy

Know how your database backs up, and verify it. Every managed database platform has automated backups — turn them on and test a restore before you have real user data.

5. Vendor lock-in awareness

Containerize from the start (even if deploying to PaaS). Use a database-agnostic ORM. Instrument with OpenTelemetry instead of Datadog's SDK.

Write down your exit path: "If we needed to leave [platform], here's how we'd migrate and roughly how long it would take." If you can't answer that, you're more locked in than you think.

On AI-generated infrastructure code:

79% of organizations are using AI to generate IaC. Only 13% feel "extremely prepared" for the security implications. AI generates syntactically correct Terraform that may put your RDS instance in a public subnet with 0.0.0.0/0 because it followed syntax patterns without security context.

Treat AI-generated IaC as a first draft. Require human review before terraform apply. Scan with Checkov or similar before every apply.
SECTION 6: Build Activity — 30 Minutes From Blank Repo (1:27–1:57)

Everyone starts a new project from scratch
The Build Loop (Frame Before You Start)

Before the activity starts, give students this three-step loop. It's the repeating cycle that replaces "code and pray":

1. Build it — let it run

    Write a prompt. Give AI the context it needs. Let it build.
    Don't over-supervise. Let the model finish before evaluating.
    Adjust your prompts if the output isn't right — don't debug AI errors, fix the prompt.
    And don't be afraid to start over. The restart instinct is a feature, not a failure.

2. Test it

    Automated spec verification — AI writes tests against the spec document (not its own code). This is the circular validation fix.
    Automated browser testing — Playwright MCP + Claude, or a separate post-implementation session
    Manual QA — actually click through the thing. AI misses things humans catch in 30 seconds.

3. Iterate on feedback

    Real feedback = users, tests, and your own eyes on the running product.
    Get it in front of someone. Even if it's just you.
    GitHub workflows with push-to-deploy (e.g., Laravel Forge, Railway deploy hooks, Vercel git integration) make iteration fast — commit, push, it's live, get feedback.

The loop isn't linear. You'll jump between steps. The goal is to spend more time in step 1 and less time stuck between 1 and 2 because you're afraid to commit.

SECTION 7: Discussion & Wrap-Up (1:57–2:25)
1:57–2:17 — Cohort Debrief

Structured debrief — these questions in order:

Round 1: What happened?

    "Who got something working? Show it off for 30 seconds."
    "Who hit a wall? What kind of wall — AI going off the rails, unclear requirements, tech stack issues?"
    "Who restarted? How many times?"

Round 2: The restart question

    "For those who restarted: was it faster than trying to debug? What triggered the restart?"
    "For those who didn't restart: was there a moment where you probably SHOULD have restarted?"
    "What made restarting feel uncomfortable? Was that discomfort justified by the outcome?"

Round 3: The mode question

    "Who went spec-first? How far did you get before you started building?"
    "Who went prompt-first? Did seeing the output change what you wanted to build?"
    "If you were going to do this for real — a project that needed to survive month two — which mode would you choose? Why?"

Round 4: Design / Testing / Deployment

    "What design decisions did you make in the first 5 minutes? Were they the right ones?"
    "Did anyone write a test before writing implementation code? Why / why not?"
    "What would it have taken to deploy what you built today? Did you think about that while building?"

Round 5: The meta question

    "What would you do differently in the first 30 minutes of your next greenfield project?"
