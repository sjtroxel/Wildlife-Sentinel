The Perception-Reality Gap (3-4 min)

The stat that still defines this moment:

The METR study from mid-2025 tested 16 experienced open-source developers on 246 real tasks. Before starting, they predicted AI would make them 24% faster. After finishing, they believed it made them 20% faster.

The actual result? 19% slower.

That's a 43-point gap between perception and reality—one of the largest ever recorded in software engineering research.

(Source: METR Study, July 2025)

Why this happens (2026 breakdown):

    The delegation ceiling: According to Anthropic's 2026 trends report, developers use AI in about 60% of their work—but can only fully delegate 0-20% of tasks. AI is a collaborator, not a replacement. It still requires constant supervision and validation.
    Review bottleneck: LinearB analyzed 8.1 million pull requests and found that AI-generated PRs wait 4.6x longer for review and have an 84% lower acceptance rate than human-written code.
    The overhead nobody talks about: The METR study found developers spent 9% of their time reviewing and cleaning AI outputs, plus 4% waiting for generations. That overhead erases gains.

The Real Stakes

The quality gap is measurable:

    66% of developers say inaccurate code suggestions are their biggest AI challenge
    45% report longer debugging times with AI-generated code
    GitClear's data shows sharp declines in code quality metrics even as AI adoption rises

What this looks like in practice:

Early studies from GitHub, Google, and Microsoft found developers completing tasks 20-55% faster. But a Bain & Company report called real-world savings "unremarkable." Why? Productivity gains at the front end get erased by downstream bottlenecks—bugs, security exposure, rework.

The Anthropic insight:

    "The organizations pulling ahead aren't removing engineers from the loop—they're making engineer expertise count where it matters most."

(Source: Eight Trends Defining How Software Gets Built in 2026)

Tools & Models
The Wrong Question

"What's the best AI coding tool?"

By the time I answer that, it'll be wrong. The landscape shifts every few months. Claude 4 in May, GPT 5 in August, Claude 4.5 in September, Gemini 3 in December, Claude Opus 4.6 literally today... and by the time you watch this recording, there's probably something new.

The right question: How do I evaluate tools so I can pick the right one for my workflow—and keep picking the right one as things change?
The Two Things That Actually Matter

Every AI coding tool has to solve the same problem: How does the AI understand your codebase well enough to make useful changes?

Two systems determine the answer:

1. Orchestration — How the tool breaks down tasks, plans work, and coordinates execution.

2. Context Management — How the tool feeds the right code into the model's limited attention window.

These matter more than which model is under the hood. A great orchestrator with smart context management will outperform a better model with dumb context management every time.
Models

When comparing models, look at:

    Context window — How much code can it "see" at once?
    Speed (TPS) — Tokens per second. Matters for iteration speed.
    Experience over benchmarks — Benchmarks lie. Test on YOUR codebase.
    Instruction following — Does it respect your AGENTS.md / CLAUDE.md / .cursorrules?

Tools (Orchestration)

When comparing tools, look at:

    Built-in agent tools — What can it do? (file ops, terminal, browser, search)
    Terminal access — Can it run commands and see results?
    Extensibility — MCP, Skills, custom agents, slash commands
    Context management — Sub-agents? Compacting? Persistent files?
    Parallel execution — Can multiple agents work at once? Can they coordinate with each other (agent teams) or only report back to a lead (sub-agents)?
    Self-verification — Does it check its own work? (lint, test, screenshot)
    Observation & Interruptability — How easy is it for you to peek in on what it's actually doing, and can you message individual agents directly?

Evaluating Orchestration (The Framework)

When you're looking at any tool, ask these six questions:

    Context gathering — How does it learn your codebase? Embeddings? Repo maps? Sub-agents?
    Context isolation — Can it explore 50 files without polluting your main conversation?
    Planning — Can it break a big task into steps before diving in?
    Parallel execution — Can multiple agents work without conflicts? Sub-agents (report back to lead) vs. agent teams (coordinate directly with each other)?
    Self-correction — Can it run tests, see errors, and fix them?
    Persistence — Does it remember what it learned between sessions?

The "Master One Tool" Argument

My take: for most dev tasks, learning one tool deeply beats juggling models.

    Orchestration is tool-specific. Learning Claude Code's sub-agent patterns compounds. Same for Cursor's agent sidebar.
    Context management is where expertise matters. Knowing when to add files, how to structure context, when to compact—this is the skill that makes "19% slower" go away.
    Model swapping is secondary. Once you've mastered orchestration, experimenting with models is low-effort.
